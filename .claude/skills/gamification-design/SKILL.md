# Gamification Design (학습앱 행동설계)

## 0. 소유 + 경계 (먼저 읽기)

- **소유**: 이 스킬은 **게임화 리드**의 것이다. "이 화면에 무엇을 왜 넣을지" 프로덕트 판단만 다룬다.
- **이 스킬이 다루지 않는 것**: UI 토큰·컬러 값·컴포넌트 구현·듀오링고 차용/변형/거절 매트릭스 검사 → 이건 **디자인 리드 + design-review/frontend-design 스킬** 담당이다. 색·라운드·그림자·애니메이션 같은 시각 실행을 여기서 결정하지 말 것.
- **이 스킬이 정하지 않는 것**: 구체적 수치(기억HP 단위, 등급/뱃지 단계 수, 리그 도입 시점, 푸시 종류·시간·횟수)는 잠긴 SSoT가 이미 정했다. 본 스킬은 "왜 그렇게 생각하는가"의 판단 프레임만 준다. **충돌 시 SSoT가 이긴다.** (→ §5 앵커 참조)
- **서비스화 가능 원칙**: 제안하는 메카닉은 실제 구현 가능성을 전제한다. 게임화 강도가 높을수록 구현 복잡도가 오르므로, 제안 시 Tech Lead와 우선순위를 협의한다.
- **그린필드 인지**: 본 프로덕트는 완전 신규 스택(Next.js 15 + FSD + Supabase)으로 레거시 제약이 없다. 대담한 메카닉 제안이 가능하되, 1인 창업 팀 규모를 고려한 우선순위 판단이 필수.

## 핵심 전제

이 제품은 "공부를 시키는 앱"이 아니라 **"매일 다시 여는 습관을 만드는 엔진"**이다. 수능 콘텐츠 질은 경쟁자도 가질 수 있다. 차별점은 retention 설계다. 모든 화면/기능 설계 시 첫 질문: **"이 화면은 코어 루프의 어느 단계인가?"**

## When to Use This Skill

- 새 화면/기능의 게이미피케이션 요소를 설계할 때
- 보상 구조(XP·뱃지·등급·streak)를 정의하거나 조정할 때
- 알림/푸시/리텐션 트리거를 설계할 때
- 기존 화면을 retention 관점에서 리뷰할 때
- "이 기능에 게임 요소를 넣을지 말지" 판단이 필요할 때

## 프레임워크 1 — Hook Model (코어 루프의 뼈대)

모든 핵심 플로우는 4단계 루프로 검증한다:

1. **Trigger(계기)** — 외부(푸시·streak 위험 알림) + 내부(지루함·시험불안 해소 욕구)
2. **Action(행동)** — 최소 마찰로 시작. 1탭으로 학습 시작 가능해야 함
3. **Variable Reward(가변 보상)** — 예측 불가능한 보상. XP 양을 고정하지 말 것
4. **Investment(투자)** — 사용자가 뭔가 쌓게 함. 다음 trigger를 강화하는 자산(streak·진도·정답노트)

> 설계 리뷰 규칙: 핵심 플로우에서 4단계 중 빠진 게 있으면 retention 누수 지점이다.

## 프레임워크 2 — Octalysis 8 Core Drives (어떤 동기를 건드릴지)

| 드라이브 | 학습앱 적용 | 수능생 타겟 우선순위 |
|---|---|---|
| Meaning(의미) | "원하는 대학" 목표 연결 | **높음** |
| Accomplishment(성취) | 진도·마스터리·레벨 | **최우선** |
| Empowerment(권한·창의) | 오답 커스텀 복습 경로 | 중 |
| Ownership(소유) | 정답노트·캐릭터·통계 자산화 | 중 |
| Social Influence(사회적) | 리그·친구 streak | 중(주의) |
| Scarcity(희소) | 한정 이벤트·잠금 콘텐츠 | 저 |
| Unpredictability(불확실) | 가변 XP·랜덤 보상 | 중 |
| Avoidance(회피·손실) | streak 손실·리그 강등 | **낮춤(의도적)** |

## 프레임워크 3 — 핵심 메카닉 4종

- **Loss Aversion**: streak 손실, 리그 강등, 기억HP 고갈. 가장 강력한 레버지만 스트레스를 유발 → 수능생엔 강도 조절
- **Variable Reward**: 레슨 완료 시 XP를 ±변동. 기대값 이상 몰입 유발
- **Visible Progress**: XP바·레벨·진도·streak 카운터를 겹쳐서 "항상 뭔가 전진 중"으로 보이게
- **Micro-learning**: 세션을 3~5분으로 끊는다. 장벽은 끝내기가 아니라 시작하기. 활성화 에너지를 0에 가깝게

## 수능생 타겟 변형 규칙 (이 스킬의 차별점)

듀오링고를 그대로 복붙하지 말 것. 수능생은 이미 시험 스트레스가 극대인 집단이다. 손실회피·중독 레버를 그대로 쓰면 윤리적 리스크 + 번아웃 + 부모/교사 반발.

**무게중심 이동: 회피(Avoidance) → 성취·숙달(Accomplishment/Mastery)**

- streak는 유지하되 "복구권(streak freeze)"을 넉넉히 → 손실 공포 대신 안심
- 리그 경쟁은 옵트인 또는 활동량 그룹 매칭으로 과열 방지
- 푸시는 죄책감 유발("돌아와!") 대신 성취 리마인드("어제 3단원 끝냈어, 오늘 4단원?")
- "오늘 충분히 했다"는 명확한 완료 신호 제공 → 무한 압박 금지
- 부모/교사가 진도를 볼 수 있는 투명성 = 신뢰 자산 (B2B2C 채널에 유리)

> 판단 기준: 어떤 메카닉이 "불안을 연료로 쓰는가?"를 물어라. 그렇다면 성취 기반 대안으로 재설계.

## 컬러 시맨틱 규칙

색은 장식이 아니라 상태 신호다(진행/성공 · XP·획득 · 잠금 · 오답·경고 · streak·연속 각각 고정 매핑). **단, 구체 토큰값·hex·Tailwind 변수는 본 스킬이 정하지 않는다.**

우리 컬러/비주얼 SSoT는 **v2 Teal/Mint 디자인 시스템**(SOO-260616-01, 2026-06-16 palette v2.0; `docs/design-system/2026-06-09-design-system-lock.md` v2.1; 팔레트 SSoT 출처 SOO-19 v3 `110076f`)이다. 토큰 정확값은 `apps/web/src/shared/styles/tokens.css`가 최종이며, 매핑은 design-review/frontend-design 스킬(디자인 리드)이 담당한다. 게임화 판단 시에는 "이 상태가 어떤 신호 색 카테고리인가"까지만 정하고, 실제 값은 디자인 리드에 위임한다. `app_UI.png` / `web_ui.png`는 시안 reference 역할만 — 픽셀 직접 채취는 폐기.

## 5. 이 프로젝트에서의 적용 (앵커 — 판단 전 반드시 확인)

> **우선순위 규칙: 잠긴 SSoT > 이 스킬.**
> 이 스킬은 "왜/어떻게 생각할지"의 판단 프레임이다. "정해진 값"은 아래 잠긴 문서가 이긴다.
> 스킬의 일반론과 잠긴 결정이 충돌하면, **잠긴 결정을 따르고 게임화 리드에게 보고**한다. 임의로 잠긴 값을 되돌리거나 미루는 제안을 하지 말 것.

### 5-1. 이미 잠긴 결정값 (출처: `게임성_기획_구조.md` v1.0 — 게임화 SSoT)

| 항목 | 잠긴 값 | 비고 |
|---|---|---|
| 기억 HP 단위 | **0–5 정수** (기본 5) | 백분율·하트로 되돌리기 금지 |
| 사용자 전체 등급 | **6단** 누적 XP (순공입문→순공러→순공대장→순공도사→순공마왕→순공전설) | 식물 모티프(씨앗/새싹 등) 회귀 금지 |
| 개념 숙련 등급 | **4단** 점수 기반 (위험/불안정/안정권/장기기억) | |
| 뱃지 희귀도 | **4단계** (일반/희귀/영웅/전설) | 3단으로 축소 금지 |
| 순공리그 | **MVP 1.5차 도입** (MVP 1차엔 sidebar 잠금 표시) | **"리그 빼자/미루자"는 회귀 — 금지** |
| 푸시 | **6종 / 시간대 3개(8·18·22시) / 하루 최대 2회 / 23시 이후 금지** | |
| 인정 순공시간 cap | 하루 **60분**까지만 리그 점수 반영 | 밤샘·타이머 뻥튀기 방지 |
| MVP 1차 게임화 7종 | 회독 3개 / XP / 스트릭 / 기억HP / 기본 등급 / 뱃지 10종 / 회독 완료 푸시 | |

→ 새 게임화 요소를 제안할 때 **위 값과 충돌하는지 먼저 대조**한다. 충돌하면 제안을 멈추고 보고.

### 5-2. 화면별 게임성 강도 (잠김)

홈 30% · 회독퀘스트 20% · 결과 50% · 오답던전 60% · 망각방어전 60% · 순공리그 70% · 4점보스 80%. "전체를 게임처럼" 만들지 말 것. 홈은 학습 앱 톤, 게임성은 결과·서브 모드에서 올린다.

### 5-3. 참조 문서 (충돌 시 우선순위 높은 순)

1. `01_제품_UX_게임화/게임성_기획_구조.md` — 게임화 값 SSoT (최우선)
2. `docs/design-system/2026-06-09-design-system-lock.md` + `apps/web/src/shared/styles/tokens.css` — 컬러·비주얼 SSoT (v2 Teal/Mint; SOO-96 baseline. 토큰 정확값이 최종). `app_UI.png` / `web_ui.png`는 시안 reference만 — 픽셀 직접 채취 폐기
3. `01_제품_UX_게임화/순공대장_UI_설계.md` — 듀오링고 차용/변형/거절 매트릭스, Light Study Garden 톤, 화면별 현행 명칭(회독퀘스트·오답던전·망각방어전·순공냅스 등)

### 5-4. 폐기·회귀 금지 (게임화 관련)

- ❌ Dark RPG / 다크 네이비 / "회독마왕" 방향
- ❌ Fear-based 카피 ("안 사면 점수 떨어진다", "1등급 못 간다" 등)
- ❌ 손실회피·중독 레버를 강하게 (수능생 번아웃 리스크)
- ❌ 잠긴 결정값(§5-1)을 임의로 변경·축소·연기

### 5-5. 마스코트

마스코트는 듀공 = **순공이**(가안)로 고정. 게으르거나 위협적 캐릭터로 바꾸지 말 것. loss 상황도 "순공이 sleep mood"처럼 차분하게 표현(같은 효과를 부드럽게).

## Output 형식

게이미피케이션 설계를 산출할 때 항상 포함:
1. 어떤 코어 드라이브를 건드리는가 (Octalysis 매핑)
2. Hook 루프 어느 단계인가
3. 어떤 메카닉을 쓰는가
4. 수능생 변형 규칙을 통과했는가 (불안 연료 체크)
5. **잠긴 SSoT(§5)와 충돌하지 않는가** — 충돌 시 보고
6. 측정 지표 (D1/D7 retention, 세션 완료율 등)

## Examples

**예: "데일리 목표 화면을 설계해줘"**
→ Octalysis: Accomplishment 중심 / Hook: Action+Investment 단계 / 메카닉: Visible Progress + Micro-learning / 변형: 완료 신호 명확화, 미달 시 죄책감 카피 금지 / SSoT 체크: 홈 게임성 30% 강도 준수, 회독 3개 구조 유지 / 지표: 일일 목표 달성률·D1 retention

**예: "리그 시스템 넣을까?"**
→ 리그는 Social+Avoidance 레버라 수능생 과열 위험이 있어 강도 조절이 핵심이다. **단, 도입 여부는 이미 잠긴 결정이다** — 순공리그는 MVP 1.5차 도입(MVP 1차엔 잠금 표시)으로 확정. "안 넣는다"는 선택지는 회귀이므로 제안하지 말 것. 게임화 리드의 역할은 *도입 여부가 아니라* 과열 완화 설계다: 활동량 매칭, 강등 페널티 완화, 점수는 행동 기반(인정 순공분 cap 60분), CTA는 "공부 더 해라"가 아니라 "오늘 회독하고 +점수".
