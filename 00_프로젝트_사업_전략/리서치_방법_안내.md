좋아. 그럼 순공대장 에이전트 구조에 **“영상 캡처 / 인강 회독화 Agent”**를 추가하면 돼.

핵심은 이거야.

> **인강 영상을 저장·복제하는 앱이 아니라, 학생이 본 인강의 핵심 장면·판서·개념을 복습 퀘스트로 바꾸는 앱**

특히 유료 인강은 저작권 리스크가 크기 때문에, 초기 MVP에서는 **자동 녹화·영상 저장·강의 전체 캡처**는 피하는 게 좋아. 교육부·문체부의 원격수업 저작물 이용 안내에서도 수업 영상·사진 캡처물을 다른 곳에 옮기면 안 된다고 주의하고 있고, 수능 기출도 평가원 허락 없는 복제·배포·전자출판을 금지한다고 명시되어 있어. ([korea.kr](https://www.korea.kr/news/policyNewsView.do?newsId=148871475&utm_source=chatgpt.com))

# 순공대장 업데이트: 영상 캡처 Agent 추가

## 1. 영상 캡처 기능의 역할

영상 캡처는 “인강을 가져오는 기능”이 아니라, **학생이 공부 중 멈춘 순간을 회독 퀘스트로 바꾸는 기능**으로 정의해야 해.

예를 들면 학생이 인강을 보다가 이런 장면에서 캡처한다.

> 수학 수열 3강 27분 12초  
> 선생님이 점화식 일반항 풀이를 설명하는 판서 장면  
> 학생 메모: “여기서 n+1이랑 n 헷갈림”

그러면 AI는 이걸 이렇게 바꾼다.

```json id="xom8s5"
{
  "source_type": "video_capture",
  "lecture_title": "수학 수열 3강",
  "timestamp": "27:12",
  "subject": "수학",
  "unit": "수열",
  "topic": "점화식",
  "detected_content": "점화식 일반항 도출",
  "student_note": "n+1과 n을 헷갈림",
  "review_type": "개념 회독",
  "quest": [
    "점화식에서 초항을 먼저 확인하라",
    "n과 n+1의 관계를 다시 써보라",
    "비슷한 확인문항 1개 풀기"
  ],
  "review_schedule": ["1일 후", "3일 후", "7일 후", "14일 후"]
}
```

즉, 영상 캡처는 **문제 생성 소스**라기보다 **복습 트리거 소스**야.

# 2. 추가되는 Agent 구조

기존 구조에 이걸 추가하면 된다.

```text id="229tk6"
[문제사진 입력 Agent]
문제집 / 오답 / 모의고사 사진 업로드

[영상 캡처 Agent]
인강 화면 캡처 / 강의명 / 타임스탬프 / 학생 메모 입력

↓
[OCR & 화면분석 Agent]
판서, 자막, 슬라이드, 공식, 키워드 추출

↓
[개념추출 Agent]
과목, 단원, 개념, 유형, 약점 포인트 태깅

↓
[오답/망각위험 Agent]
학생이 헷갈린 지점, 다시 봐야 할 개념 추론

↓
[복습퀘스트 Agent]
개념 카드, 빈칸문제, OX문제, 미니문항 생성

↓
[회독 Agent]
1일/3일/7일/14일 복습 예약

↓
[게임 Agent]
오늘의 회독, 망각방어전, XP, 스트릭, 순공리그 반영
```

# 3. 영상 캡처는 3가지 모드로 나눠야 한다

## A. MVP 모드: 수동 캡처 + 학생 메모

가장 현실적인 방식.

학생이 직접 캡처 이미지를 올리거나, 앱 안에서 “회독 저장” 버튼을 누른다.

입력값은 이 정도면 돼.

| 입력 | 예시 |
|---|---|
| 강의명 | 현우진 수열 3강 |
| 과목 | 수학 |
| 단원 | 수열 |
| 캡처 이미지 | 판서 화면 |
| 타임스탬프 | 27:12 |
| 학생 메모 | “여기서 왜 등차수열이 되는지 모르겠음” |
| 이해도 | 이해함 / 애매함 / 모르겠음 |

AI는 이걸 바탕으로 복습 카드를 만든다.

예:

> 순공이: “이 개념 애매하다고 저장했네! 오늘 점화식 망각방어전 같이 가보자.”

---

## B. 안전 모드: 원본 캡처는 오래 저장하지 않기

저작권 리스크를 줄이려면, 서버에는 원본 캡처 이미지를 장기 저장하지 않는 방식이 좋다.

추천 구조는 이거야.

```text id="blb242"
학생 캡처 업로드
→ OCR/분석
→ 개념 태그와 복습 카드 생성
→ 원본 이미지는 로컬 보관 또는 일정 시간 후 삭제
→ 서버에는 파생 데이터만 저장
```

서버에 남기는 건 이런 데이터야.

```json id="rocaqg"
{
  "lecture_title": "수학 수열 3강",
  "timestamp": "27:12",
  "subject": "수학",
  "unit": "수열",
  "topic": "점화식",
  "concept_summary": "점화식에서 일반항을 구하는 과정",
  "student_confusion": "n과 n+1 관계 혼동",
  "review_card": "점화식 문제에서는 초항과 인덱스를 먼저 확인한다."
}
```

이렇게 하면 앱의 본질이 **강의 저장소**가 아니라 **개인 복습 기록장**이 된다.

---

## C. 제휴 모드: 인강사와 계약 후 강의 메타데이터 연동

장기적으로는 이게 가장 강력해.

인강 업체와 제휴하면 학생이 강의를 본 기록을 가져올 수 있어.

예:

| 데이터 | 활용 |
|---|---|
| 강의명 | 복습 퀘스트 생성 |
| 단원명 | 개념 태깅 |
| 시청 시간 | 순공/회독시간 구분 |
| 완강 여부 | 복습 필요도 계산 |
| 강의별 핵심 개념 | 개념 카드 자동 생성 |
| 교재 문제 ID | 오답던전 연결 |

이 단계가 되면 순공대장은 이렇게 말할 수 있어.

> “인강을 봤는지”가 아니라  
> **“인강에서 배운 걸 다시 회수했는지”를 관리한다.**

# 4. 영상 캡처 Agent가 만들어야 하는 산출물

영상 캡처 1개가 들어오면 바로 문제를 만들기보다, 아래 5개를 만들어야 해.

## 1) 개념 카드

```text id="7unkfr"
점화식 문제에서는 a₁, a₂ 같은 초기 조건과 n, n+1의 관계를 먼저 확인해야 한다.
```

## 2) 빈칸 회독

```text id="3gecz7"
점화식 a_{n+1}=a_n+2n 에서 일반항을 구할 때 먼저 확인해야 하는 값은 ____이다.
```

## 3) OX 확인

```text id="qwspxu"
점화식 문제에서는 n과 n+1의 위치가 바뀌어도 결과는 같다. O / X
```

## 4) 미니문항

```text id="2i8nu8"
a₁=1, a_{n+1}=a_n+2n일 때 a₃의 값은?
```

## 5) 망각방어 퀘스트

```text id="0tp3m4"
3일 후: 같은 개념의 짧은 문제 1개
7일 후: 원래 캡처한 개념 다시 설명하기
14일 후: 유사문항 1개 풀기
```

# 5. 영상 캡처 DB 구조 추가

기존 문제 DB에 아래 테이블을 추가하면 돼.

## `lecture_capture_source`

| 필드 | 설명 |
|---|---|
| capture_id | 캡처 ID |
| user_id | 학생 ID |
| source_type | 인강 캡처 / 직접 노트 / 제휴 강의 |
| provider_name | 메가스터디, 대성, 이투스 등 |
| lecture_title | 강의명 |
| timestamp | 영상 위치 |
| capture_image_url | 원본 캡처 이미지 |
| storage_policy | 로컬 / 임시저장 / 제휴허용 |
| copyright_status | 개인학습용 / 제휴허용 / 노출불가 |
| student_note | 학생 메모 |

## `lecture_capture_analysis`

| 필드 | 설명 |
|---|---|
| capture_id | 캡처 ID |
| subject | 과목 |
| unit | 단원 |
| topic | 세부 개념 |
| detected_text | OCR 추출 텍스트 |
| concept_summary | 개념 요약 |
| confusion_point | 헷갈린 지점 |
| quest_type | 개념회독 / 오답던전 / 망각방어 |
| review_schedule | 1일/3일/7일/14일 |

## `lecture_review_quest`

| 필드 | 설명 |
|---|---|
| quest_id | 퀘스트 ID |
| capture_id | 원천 캡처 |
| user_id | 학생 |
| quest_format | 개념카드 / OX / 빈칸 / 미니문항 |
| due_date | 복습 날짜 |
| result | 성공 / 실패 / 스킵 |
| confidence | 학생 자신감 |

# 6. 영상 캡처 기능의 MVP 화면 흐름

초기 앱 화면은 이렇게 단순하게 가면 돼.

```text id="nit5i3"
[오늘 본 인강 입력]
과목: 수학
강의명: 수열 3강
오늘 본 시간: 42분

[헷갈린 장면 캡처]
이미지 업로드 또는 캡처 저장

[학생 메모]
"점화식 일반항 구하는 부분이 헷갈림"

[AI 분석]
과목: 수학
단원: 수열
유형: 점화식
오답위험: n/n+1 인덱스 혼동

[오늘의 회독 생성]
1. 개념 카드 1개
2. OX 문제 1개
3. 미니문항 1개
4. 3일 후 망각방어 예약
```

# 7. 순공대장 전체 입력 소스 업데이트

이제 순공대장의 입력 소스는 4개가 된다.

| 입력 소스 | 목적 | MVP 우선순위 |
|---|---|---|
| 문제사진 | 오답/문항 분석 | 최우선 |
| 인강명/단원명 | 복습 퀘스트 생성 | 최우선 |
| 영상 캡처 | 헷갈린 개념 회독화 | 최우선 |
| 기출/자체제작 문제 | 유사문항 생성 | 2단계 |

이 구조가 좋다.  
왜냐하면 학생 입장에서 실제 학습 흐름이 이렇기 때문이야.

```text id="lzup23"
인강을 본다
→ 헷갈린 장면을 캡처한다
→ 문제를 푼다
→ 틀린 문제를 찍는다
→ AI가 오늘/내일/3일 뒤 다시 풀게 한다
```

이게 순공대장의 핵심 루프가 된다.

# 8. 법적·운영 리스크를 줄이는 원칙

서비스 문구는 이렇게 잡는 게 안전하다.

> 순공대장은 인강 영상을 저장하거나 배포하지 않습니다.  
> 학생이 개인 학습 목적으로 입력한 강의명, 메모, 캡처를 분석하여 복습 퀘스트를 생성합니다.  
> 유료 강의 원본 영상, 전체 캡처, 강의자료의 무단 공유 기능은 제공하지 않습니다.

기술적으로도 아래 원칙이 필요해.

| 위험 기능 | 대체 방식 |
|---|---|
| 강의 전체 녹화 | 금지 |
| 자동 캡처 반복 저장 | MVP 제외 |
| 유료 강의 자막 전체 추출 | 제휴 전 금지 |
| 캡처 이미지 커뮤니티 공유 | 금지 |
| 원본 이미지 장기 서버 저장 | 최소화 |
| 판서/문제 원문 그대로 재배포 | 금지 |
| 개념 태그/복습카드 생성 | 허용 방향 |
| 학생 개인 메모 기반 퀘스트 | 허용 방향 |

# 9. 업데이트된 한 줄 정의

기존:

> 순공대장은 문제를 풀어주는 AI가 아니라, 다시 풀게 만드는 AI다.

영상 캡처까지 넣으면 이렇게 바뀐다.

> **순공대장은 인강·문제사진·오답을 AI가 회독 퀘스트로 바꿔, 학생이 까먹기 전에 다시 공부하게 만드는 수능 반복학습 게임이다.**

또는 더 강하게:

> **인강 보고 끝? 캡처하면 순공이가 다시 물어보러 와.**

# 10. 최종 MVP 개발 범위 업데이트

내가 추천하는 1차 MVP는 이거야.

```text id="53gwkz"
1. 학생 회원가입
2. 문제사진 업로드
3. 인강명/단원명 입력
4. 영상 캡처 1장 업로드
5. 학생 메모 입력
6. AI 과목/단원/개념 태깅
7. AI 오답/헷갈림 원인 분류
8. 개념카드/OX/빈칸/미니문항 생성
9. 1일/3일/7일/14일 복습 예약
10. 오늘의 회독
11. XP/레벨/스트릭
12. 순공이 피드백
```

즉, 순공대장의 초기 핵심은 이것.

> **문제사진 + 영상캡처 + 학생메모 → 오늘의 회독**

이러면 “수능 기출 문제를 대량으로 만들어야 하는 앱”이 아니라,  
**학생이 실제 공부한 흔적을 매일 복습하게 만드는 AI 리텐션 앱**으로 포지셔닝할 수 있어.
