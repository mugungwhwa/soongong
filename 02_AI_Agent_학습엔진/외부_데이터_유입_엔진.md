# 외부 데이터 유입 엔진

> 원본 소스: `외부 데이터 유입 엔진.txt`  
> 분류: **AI Agent·학습 엔진**

---

좋아. 순공대장에서는 **외부 데이터 유입 엔진**을 이렇게 정의하면 돼.

> **외부 데이터 유입 엔진 = 문제사진, 인강기록, 캡처, 교재/기출 유형 정보, 제휴 콘텐츠, 학생 풀이반응을 받아서 “회독 가능한 학습 객체”로 정제하는 엔진**

즉, 단순 크롤러가 아니라 **Source → Analysis → RAG → Quest**로 바꾸는 입구야.

---

# 1. 핵심 결론

순공대장 외부 데이터 유입 엔진은 이렇게 가야 해.

```text id="20g1nl"
외부 데이터
문제사진 / 인강기록 / 캡처 / 메모 / 제휴 문제 / 유형카드 / 학생 풀이로그

↓
Data Intake Engine
권한 확인 / 저작권 상태 분류 / OCR / 구조화 / 중복 제거 / 메타데이터 추출

↓
Learning Object DB
문제 원문 DB가 아니라 학습 객체 DB

↓
RAG
Curriculum RAG / Type Pattern RAG / Student Memory RAG

↓
Agent
문항분석 / 오답진단 / 난이도 / 베리에이션 / 회독스케줄 / 퀘스트 생성

↓
오늘의 회독
오답던전 / 망각방어전 / 순공리그 / XP
```

이미 순공대장 백서 방향에서도 16개 Agent, 3층 RAG, Supabase 기반 데이터 파이프라인, `learning_sources`, `type_pattern_cards`, `review_quests`, `solve_events` 같은 DB 구조를 잡아뒀기 때문에 이 유입 엔진은 그 앞단에 붙는 **데이터 게이트웨이**로 보면 된다. fileciteturn2file0

---

# 2. 외부 데이터는 5종류로 나눈다

## 1) 학생 생성 데이터

가장 중요한 MVP 데이터.

| 데이터 | 예시 | 역할 |
|---|---|---|
| 문제사진 | 문제집, 오답, 모의고사 사진 | 실제 오답 분석 |
| 인강 기록 | “수학 수열 3강” | 복습 카드 생성 |
| 영상 캡처 | 판서/개념 캡처 1장 | 헷갈린 개념 회독화 |
| 학생 메모 | “n+1이 헷갈림” | 오답 원인 추론 |
| 정답/오답 입력 | 맞음/틀림 | 숙련도 업데이트 |

초기 MVP는 **학생이 직접 올린 문제사진과 인강/메모 입력**이 중심이어야 한다. 기존 설계에서도 학생 문제사진과 인강/단원명 입력을 MVP 최우선 입력으로 잡았고, 공개 기출 원문은 내부 연구·태깅 기준용으로만 조심해서 다루는 방향이 맞다고 정리되어 있다. fileciteturn2file2

---

## 2) 제휴/라이선스 데이터

2단계 이후.

| 데이터 | 예시 | 역할 |
|---|---|---|
| 라이선스 문제 | 출판사/강사/학원 제휴 문제 | 안전한 문제은행 |
| 자체 제작 문제 | 내부 제작 수학/탐구 문항 | 유사문항 생성 |
| 교재 메타데이터 | 단원, 유형, 난이도 | 유형 매칭 |
| 강의 메타데이터 | 강의명, 단원, 핵심개념 | 인강 회독화 |

여기서 핵심은 **원문 사용 권한이 명확한 데이터만 서비스에 노출**하는 거야.

---

## 3) 수능 유형 지식 데이터

순공대장의 기술 해자.

| 데이터 | 예시 | 역할 |
|---|---|---|
| 교육과정 개념표 | 수학 > 수열 > 점화식 | Curriculum RAG |
| 수능 유형 카드 | 점화식 특정항 계산형 | Type Pattern RAG |
| 오답 패턴 | 조건 누락, 인덱스 혼동 | 오답진단 |
| 난이도 요인 | 조건 수, 계산량, 개념 결합 | Difficulty Agent |
| 변형 가능 축 | 숫자, 요구값, 표현방식 | Variation Agent |

중요한 건 **기출 원문 RAG가 아니라 수능 유형 카드 RAG**라는 점이다. LLM만 쓰면 유형 일관성, 교육과정 정합성, 난이도 판단, 저작권 리스크가 흔들릴 수 있어서 RAG가 기준표 역할을 해야 한다. fileciteturn2file1

---

## 4) 학생 반응 데이터

난이도 Agent와 회독 스케줄의 연료.

| 데이터 | 의미 |
|---|---|
| 정답 여부 | 기본 성과 |
| 풀이 시간 | 너무 쉬운지, 막혔는지 |
| 힌트 사용 | 자력 해결 여부 |
| 재시도 횟수 | 막힘 정도 |
| 해설 확인 | 개념 미흡 가능성 |
| 자신감 입력 | 체감 숙련도 |
| 스킵 | 과부하 신호 |
| 같은 유형 재오답 | 망각/개념 결손 |

순공대장 난이도 설계에서도 최근 5개 같은 개념/유형 문제 기준으로 정답률, 풀이시간, 힌트 사용률, 재오답률, 자신감을 보고 숙련도를 판단하는 구조가 제안되어 있다. fileciteturn1file5

## 5) 운영/검수 데이터

교육 서비스라서 반드시 필요.

| 데이터 | 역할 |
|---|---|
| 관리자 검수 결과 | 승인 / 수정 / 폐기 |
| 문제 오류 신고 | 품질 개선 |
| 저작권 위험 플래그 | 원문 유사도 관리 |
| AI 생성물 로그 | 책임 추적 |
| Agent 실행 로그 | 디버깅/품질관리 |

# 3. 엔진 이름은 이렇게 잡으면 좋다

내 추천은 이거야.

> **Source-to-Quest Engine, SQE**

한국어로는:

> **학습소스 회독화 엔진**

왜냐하면 이 엔진의 목적은 외부 데이터를 저장하는 게 아니라 **회독 퀘스트로 바꾸는 것**이기 때문이야.

# 5. 핵심 DB 설계

## A. `external_sources`

모든 외부 유입의 원장.

| 필드 | 설명 |
|---|---|
| source_id | 원천 ID |
| user_id | 학생 ID |
| source_type | problem_photo / lecture_log / capture_note / licensed_question / type_card / solve_event |
| provider_type | user / partner / internal / public_reference |
| provider_name | 학생, 제휴사, 자체제작 등 |
| license_status | user_private / licensed / internal_only / forbidden / needs_review |
| storage_policy | permanent / temporary / derived_only / local_only |
| raw_url | 원본 위치 |
| raw_text | 입력 텍스트 |
| source_hash | 중복 제거용 |
| created_at | 유입 시각 |

---

## B. `source_compliance_checks`

저작권/개인정보/노출 가능 여부.

| 필드 | 설명 |
|---|---|
| check_id | 검사 ID |
| source_id | 원천 ID |
| copyright_risk | low / medium / high |
| contains_paid_lecture | true / false |
| contains_exam_original | true / false |
| contains_personal_info | true / false |
| allow_user_view | true / false |
| allow_ai_generation | true / false |
| allow_rag_indexing | true / false |
| action | pass / redact / derived_only / reject / admin_review |

여기서 중요한 정책은:

```text id="nuybwk"
원본 저장 가능 여부와
파생 데이터 저장 가능 여부를 분리한다.
```

예를 들어 인강 캡처는 원본 이미지를 장기 저장하지 않고, OCR/개념태그/학생메모 기반 파생 데이터만 저장하는 방식이 안전하다.

---

## C. `parsed_learning_objects`

외부 데이터를 학습 객체로 바꾼 결과.

| 필드 | 설명 |
|---|---|
| object_id | 학습 객체 ID |
| source_id | 원천 ID |
| object_type | question / concept_note / lecture_concept / wrong_answer / type_pattern |
| subject | 과목 |
| unit | 대단원 |
| topic | 세부개념 |
| question_type | 문항유형 |
| difficulty_level | L1~L5 |
| extracted_text | OCR/정제 텍스트 |
| student_note | 학생 메모 |
| detected_wrong_reason | 오답 원인 |
| review_priority | low / medium / high |
| confidence_score | AI 분석 신뢰도 |

---

## D. `type_pattern_cards`

수능 유형 RAG의 핵심.

| 필드 | 설명 |
|---|---|
| type_id | 유형 ID |
| subject | 과목 |
| unit | 단원 |
| topic | 세부개념 |
| type_name | 유형명 |
| stem_structure | 문제 구조 |
| cognitive_skill | 요구 사고 |
| common_mistakes | 자주 틀리는 원인 |
| variation_axes | 변형 가능 축 |
| difficulty_factors | 난이도 상승/하락 요인 |
| validation_rules | 검증 규칙 |
| copyright_safety_rules | 원문 유사도 금지 조건 |
| embedding | pgvector 검색용 |

Type Pattern RAG는 원문 기출을 넣는 곳이 아니라, **수능 기출 유형을 추상화한 카드**가 들어가는 곳이다. 기존 설계에서도 Curriculum RAG, Type Pattern RAG, Student Memory RAG의 3층 구조를 권장했고, Type Pattern RAG는 문제 구조·표현 방식·요구 사고·오답·변형 가능 축을 관리하는 역할로 정리되어 있다. fileciteturn2file1

---

## E. `student_memory_items`

학생 개인 RAG.

| 필드 | 설명 |
|---|---|
| memory_id | 메모리 ID |
| user_id | 학생 ID |
| object_id | 연결 학습 객체 |
| concept_key | 예: math_sequence_recurrence |
| wrong_reason | 인덱스 혼동 |
| mastery_score | 0~100 |
| recent_accuracy_5 | 최근 5개 정답률 |
| hint_rate_5 | 최근 힌트 사용률 |
| confidence_avg | 자신감 평균 |
| next_review_at | 다음 회독일 |
| forgetting_risk | low / medium / high |

# 6. 유입 엔진의 Agent 구성

## 1) Source Intake Agent

역할: 외부 데이터가 들어오면 먼저 분류.

```json id="7k9m0h"
{
  "source_type": "problem_photo",
  "provider_type": "user",
  "license_status": "user_private",
  "storage_policy": "temporary_raw_then_derived",
  "next_agent": "OCR_Parsing_Agent"
}
```

---

## 2) Compliance Gate Agent

역할: 이 데이터를 저장/분석/노출/RAG화해도 되는지 판단.

판단 기준:

| 상황 | 처리 |
|---|---|
| 학생 개인 문제사진 | 개인 학습용 분석 가능 |
| 문제집 원문 대량 업로드 | 노출/공유 금지, 검토 |
| 유료 인강 캡처 | 원본 장기 저장 최소화 |
| 유료 인강 전체 녹화 | 금지 |
| 기출 원문 | 내부 연구용, 서비스 노출 주의 |
| 제휴 문제 | 계약 범위 내 사용 |

---

## 3) OCR & Parsing Agent

역할: 이미지/캡처를 구조화.

출력:

```json id="esjm94"
{
  "stem": "...",
  "choices": ["① ...", "② ...", "③ ...", "④ ...", "⑤ ..."],
  "conditions": ["n은 자연수", "a1=1"],
  "figures": ["graph_1"],
  "student_marks": ["circled_choice_3", "underline_condition"],
  "handwritten_solution": "..."
}
```

---

## 4) Learning Object Builder Agent

역할: 원천 데이터를 학습 객체로 변환.

```json id="ecfku9"
{
  "subject": "수학",
  "unit": "수열",
  "topic": "점화식",
  "question_type": "특정항 계산",
  "difficulty_level": "L3",
  "wrong_reason_candidate": ["n/n+1 인덱스 혼동", "초항 누락"],
  "review_priority": "high"
}
```

---

## 5) RAG Router Agent

역할: 어느 RAG에 넣을지 결정.

| 데이터 | 저장 위치 |
|---|---|
| 교육과정 개념 | Curriculum RAG |
| 수능 유형 카드 | Type Pattern RAG |
| 학생 오답/메모 | Student Memory RAG |
| 제휴 문제 메타 | Type Pattern + Licensed Question DB |
| 풀이 반응 | Student Memory + Mastery Table |

---

```json id="atus27"
    "xp": 40,

  }

```

# 7. 데이터 유입 정책: Raw / Derived 분리

이게 제일 중요해.

```text id="iahikp"
Raw Data = 원본 이미지, 원문, 캡처
Derived Data = 과목, 단원, 유형, 오답원인, 난이도, 복습카드, 유형카드
```

순공대장은 **Raw Data를 많이 쌓는 서비스가 아니라 Derived Data를 쌓는 서비스**가 되어야 해.

| 데이터 | 원본 저장 | 파생 저장 |
|---|---:|---:|
| 학생 문제사진 | 제한적 가능 | 가능 |
| 인강 캡처 | 임시 저장 권장 | 가능 |
| 유료 인강 영상 | 금지 | 강의명/메모 기반 가능 |
| 기출 원문 | 서비스 노출 금지 방향 | 유형 카드 가능 |
| 제휴 문제 | 계약 범위 내 가능 | 가능 |
| 학생 풀이로그 | 가능 | 가능 |

# 8. MVP 구현 범위

처음부터 외부 데이터 커넥터를 크게 만들면 안 돼.

## MVP 1차

```text id="ddifdy"
1. 문제사진 업로드
2. 인강명/단원명 입력
3. 캡처+메모 1장 입력
4. Source Registry 저장
5. OCR/텍스트 추출
6. 과목/단원/유형 태깅
7. 오답 원인 추천
```

# 10. Supabase 기준 구현

초기에는 Supabase로 충분해.

```text id="vu8693"
Supabase Auth
학생/관리자 계정

Supabase Storage
문제사진/캡처 임시 저장

PostgreSQL
학습 객체, 회독 퀘스트, 풀이 로그 저장

pgvector
Type Pattern RAG / Student Memory 검색

Edge Functions
OCR 호출, LLM 호출, Agent Orchestration

RLS
학생별 데이터 접근 제한

Cron
매일 오늘의 회독 생성
```

기존 백서에서도 Supabase 기반으로 PostgreSQL, pgvector, Storage, Edge Functions, RLS, Cron을 사용하는 데이터 파이프라인을 잡아뒀다. fileciteturn2file0

# 11. 개발 우선순위

| 기능 | 이유 |

| 완전 자동 출제 | 검수 전 위험 |

# 12. 한 줄로 정리

> **순공대장 외부 데이터 유입 엔진은 외부 문제를 많이 긁어오는 크롤러가 아니라, 학생이 만든 학습 흔적과 안전한 제휴/유형 데이터를 받아 “원문은 통제하고, 파생 학습 객체만 회독 퀘스트로 전환하는 Source-to-Quest Engine”이다.**

내가 보기엔 MVP에서는 이렇게만 만들면 된다.

```text id="00ydex"
문제사진 / 인강기록 / 캡처+메모
→ Source Registry
→ Compliance Gate
→ OCR & Parsing
→ Learning Object
→ Student Memory RAG
→ 오늘의 회독
```

이게 순공대장의 첫 번째 진짜 엔진이야.
