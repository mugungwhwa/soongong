# 과목 라우팅 시스템

> 원본 소스: `과목 라우팅 시스템.txt`  
> 분류: **AI Agent·학습 엔진**

---

좋아. **과목 라우팅은 순공대장 Agent의 첫 번째 교통정리 시스템**으로 잡으면 돼.

핵심은 이거야.

> **입력된 문제사진/인강기록/캡처/메모를 보고  
> 수학·국어·영어·탐구 중 어디로 보낼지 판단하고,  
> 그 과목에 맞는 OCR, RAG, 오답진단, 난이도, 검증 Agent를 선택하는 Router.**

기존 백서에서도 16개 Agent의 첫 단계가 **Input Router**이고, 이후 OCR, Question Analysis, Type RAG, Difficulty, Solver, Review Scheduling, Quest로 이어지는 구조로 잡혀 있어. fileciteturn2file0

---

# 1. 과목 라우팅 위치

전체 흐름은 이렇게 잡으면 된다.

```text id="jxpurd"
문제사진 / 인강기록 / 캡처 / 메모 입력

↓
[Input Router Agent]
입력 타입 분류

↓
[Subject Routing Agent]
과목 판단: 수학 / 국어 / 영어 / 탐구 / 미확정

↓
[Subject-specific Agent]
수학 Agent
국어 Agent
영어 Agent
탐구 Agent

↓
[Curriculum RAG / Type Pattern RAG / Student Memory RAG]

↓
[오늘의 회독 / 오답던전 / 망각방어전]
```

즉, **Input Router는 입력 소스를 나누고**, **Subject Router는 과목을 나눈다.**

외부 데이터 유입 엔진도 원래 `Source → Analysis → RAG → Quest` 구조로 잡았으니까, 과목 라우팅은 `Analysis` 직전 또는 초입에 들어가는 게 맞아. fileciteturn2file1

---

# 2. 과목 라우팅 Agent가 봐야 할 입력값

```json id="jivmyp"
{
  "source_type": "problem_photo",
  "user_subject_hint": "수학",
  "ocr_text": "수열 {a_n}이 a_1=1, a_{n+1}=...",
  "image_features": ["formula", "graph", "choices"],
  "lecture_title": null,
  "student_note": "점화식이 헷갈림",
  "user_history": {
    "recent_subjects": ["수학", "영어"],
    "recent_units": ["수열", "함수"]
  }
}
```

라우팅은 OCR 텍스트만 보면 안 돼.  
학생이 직접 선택한 과목, 강의명, 메모, 이미지 특징, 최근 학습 이력까지 같이 봐야 해.

---

# 3. 과목별 라우팅 기준

| 과목 | 주요 신호 | 보내야 할 Agent |
|---|---|---|
| **수학** | 수식, 그래프, 함수, 수열, 확률, 미적분 기호 | Math Analysis Agent, Math Solver, Math Type RAG |
| **국어** | 긴 지문, 보기, 선지 판단, 문학/독서/화작 키워드 | Korean Reading Agent, Evidence Agent |
| **영어** | 영어 지문, 빈칸, 순서, 삽입, 어휘, 문법 | English Passage Agent, Vocabulary Agent |
| **사탐** | 윤리, 사회문화, 경제, 정치, 역사 개념 | Social Studies Concept Agent |
| **과탐** | 물리/화학/생명/지구과학, 자료/그래프/실험 | Science Data Interpretation Agent |
| **미확정** | OCR 불량, 과목 혼재, 캡처만 있음 | Manual Confirm Flow |

# 5. Confidence 기준

라우팅 확신도에 따라 UI 처리를 다르게 해야 해.

| confidence | 처리 |
|---:|---|
| **0.85 이상** | 자동 라우팅 |
| **0.60~0.85** | AI 분석 결과 화면에서 수정 가능 |
| **0.60 미만** | 사용자에게 과목 선택 요청 |
| **OCR 실패** | “과목 먼저 선택 → 재분석” |
| **복합 과목** | 캡처/메모 기준으로 주 과목 선택 |

예시 UI:

```text id="12vrqu"
AI가 이렇게 분석했어요.

과목: 수학
단원: 수열
유형: 점화식

맞나요?
[맞아요] [수정하기]
```

# 6. 과목별 이후 처리 방식

## A. 수학 라우팅

```text id="0qw0bv"
수학 감지
→ 단원/유형 태깅
→ Type Pattern RAG 검색
→ Solver 2개 검증
→ 난이도 L1~L5 판단
→ 회독/유사문항 생성
```

수학은 MVP에서 가장 먼저 고도화해도 된다.  
템플릿 생성, 변수 치환, Solver 검증이 비교적 가능하기 때문.

---

## B. 국어 라우팅

```text id="f6pnma"
국어 감지
→ 독서/문학/화작 분류
→ 지문 원문 저장 위험 체크
→ 근거 찾기/선지 판단 훈련 중심
→ 긴 유사 지문 생성은 후순위
```

국어는 바로 문제 생성보다 **근거 문장 찾기, 선지 판단, 지문 구조 회독** 중심으로 가야 해.

---

## C. 영어 라우팅

```text id="lbw3x0"
영어 감지
→ 빈칸/순서/삽입/어휘/문법 분류
→ 짧은 자체 생성 지문 또는 개념카드
→ 원문 지문 장기 저장 최소화
```

영어는 긴 지문 원문 재사용보다, 어휘·문장 구조·짧은 미니문항으로 시작하는 게 안전해.

---

## D. 탐구 라우팅

```text id="bo3stu"
탐구 감지
→ 사탐/과탐 분류
→ 과목 세부 분류
→ 개념 비교 / 자료 해석 / 사례 적용으로 회독화
```

탐구는 수학 다음으로 MVP에 좋다.  
특히 생윤, 사문, 생명, 지구과학처럼 개념 태깅과 오답 원인 분류가 잘 맞는다.

# 7. DB 테이블 추가

기존 DB에 이 테이블 하나를 추가하면 좋다.

## `subject_routing_results`

| 필드 | 설명 |
|---|---|
| routing_id | 라우팅 ID |
| source_id | 원천 데이터 ID |
| user_id | 학생 ID |
| source_type | problem_photo / lecture_log / capture_note |
| detected_subject | 수학 / 국어 / 영어 / 탐구 |
| subject_confidence | 0~1 |
| subject_group | math / korean / english / social / science |
| unit_candidates | 단원 후보 JSON |
| topic_candidates | 세부개념 후보 JSON |
| recommended_agents | 다음 Agent 목록 |
| needs_user_confirmation | 사용자 확인 필요 여부 |
| user_corrected_subject | 사용자가 수정한 과목 |
| final_subject | 최종 과목 |
| created_at | 생성 시각 |

이후 `parsed_learning_objects`에는 최종적으로 확정된 `subject`, `unit`, `topic`, `question_type`, `difficulty_level`이 들어가면 된다. 기존 설계에서도 학습 객체 테이블은 과목, 대단원, 세부개념, 문항유형, 난이도, 오답 원인, 분석 신뢰도를 저장하는 구조였어. fileciteturn2file1

# 8. MVP에서는 이렇게만 만들면 된다

처음부터 완벽한 과목 라우팅은 필요 없어.

## MVP 1차 라우팅

```text id="ts26h4"
1. 사용자가 과목을 선택한다.
2. AI가 OCR/메모로 과목을 재확인한다.
3. 불일치하면 사용자에게 수정 요청한다.
4. 수학/영어/탐구는 자동 태깅 우선.
5. 국어는 분석만 하고 문제 생성은 제한한다.
```

즉, 초기에는 **AI 자동 판단 + 사용자 수정 가능** 구조가 제일 안전해.

# 9. 한 줄 정의

> **과목 라우팅 Agent는 순공대장에 들어온 모든 학습 흔적을 수학·국어·영어·탐구별 분석 파이프라인으로 보내고, 그 과목에 맞는 RAG·오답진단·난이도·검증 방식을 선택하는 교통정리 Agent다.**

내가 보기엔 이름은 이렇게 가면 좋아.

```text id="yuroh8"
Subject Routing Agent
한국어명: 과목 라우팅 Agent
역할명: 학습소스 과목분기 엔진
```

MVP 개발 순서는:

```text id="qoeq2u"
Input Router
→ Subject Routing Agent
→ OCR & Parsing
→ Learning Object Builder
→ RAG Router
→ Quest Conversion
```

이 순서가 가장 깔끔해.
